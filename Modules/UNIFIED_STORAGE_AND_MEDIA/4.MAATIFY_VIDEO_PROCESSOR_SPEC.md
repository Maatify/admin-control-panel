# MAATIFY Video Processor Specification

## 1. Purpose

`Maatify\VideoProcessor` is the dedicated module responsible for validating and preparing video files for final storage in version 1.

Its responsibility is intentionally narrow.

It receives a normalized uploaded file source and externally supplied video validation rules, validates the file using deterministic policy checks, prepares a temporary output artifact, and returns a formal process result DTO that can be handed to `Maatify\StorageHandler`.

This module is intentionally simpler than `Maatify\ImageProcessor` in version 1.

It does **not** perform transcoding, thumbnail generation, poster generation, streaming package generation, or codec normalization in v1.

---

## 2. Version 1 Scope

The scope of `VideoProcessor` in version 1 is deliberately restricted.

### 2.1 Included in v1

* uploaded video validation
* extension validation
* MIME validation
* max file size validation
* source readability validation
* prepared temporary output generation
* lightweight metadata extraction when safe and available

### 2.2 Excluded from v1

* transcoding
* ffmpeg-required workflows
* poster generation
* preview thumbnail extraction
* HLS/DASH generation
* bitrate normalization
* codec transformation
* multi-output workflows

---

## 3. Responsibilities

`VideoProcessor` is responsible for the following.

### 3.1 Video input validation

Validate that the uploaded file satisfies the supplied video rules.

### 3.2 Prepared temporary artifact generation

Produce a temporary prepared file suitable for handoff to `StorageHandler`.

### 3.3 Metadata normalization

Return basic metadata about the validated video when such metadata is lightweight and safely obtainable.

### 3.4 Deterministic failure behavior

Reject unsupported, corrupt, unreadable, or oversized files through exceptions.

---

## 4. Explicit Non-Responsibilities

`VideoProcessor` must **not** do any of the following in v1.

### 4.1 No final storage

It must not perform local or remote final persistence.

### 4.2 No target path generation

It must not build business folder structures or final target names.

### 4.3 No direct database dependency

It must not read any table or persistence source directly.

### 4.4 No transcoding

It must not re-encode video files in version 1.

### 4.5 No poster or thumbnail generation

It must not derive poster frames or thumbnails in version 1.

### 4.6 No streaming packaging

It must not generate segmented playback formats such as HLS or DASH.

---

## 5. Architectural Position

`VideoProcessor` sits between raw uploaded input and final storage.

### Flow

1. The consuming application receives a PSR-7 uploaded file.
2. The application builds `VideoRulesDTO`.
3. The application sends `ProcessVideoCommandDTO` to `VideoProcessor`.
4. `VideoProcessor` validates the input and prepares a temporary output file.
5. The application selects a final storage path and storage policy.
6. The application passes the prepared file to `StorageHandler`.

This module therefore owns **video preparation**, not **video persistence**.

---

## 6. Input Model

`VideoProcessor` consumes normalized DTOs.

## 6.1 `ProcessVideoCommandDTO`

### Minimum fields

* `source`: `UploadedSourceDTO`
* `rules`: `VideoRulesDTO`
* `tempDirectory`: `string|null`

### Purpose

Carry the uploaded source file and the supplied validation rules into the processor.

---

## 6.2 `UploadedSourceDTO`

The uploaded source DTO wraps the original uploaded file and informational request metadata.

### Important rule

Client-provided MIME values remain informational only and must not be treated as authoritative validation results.

---

## 6.3 `VideoRulesDTO`

The video rules DTO carries externally supplied policy values.

### Minimum fields in v1

* `allowedExtensions: array<int,string>|null`
* `allowedMimeTypes: array<int,string>|null`
* `maxSizeBytes: int|null`

### Architectural rule

This DTO may come from configuration, another module, or application-level rules. The processor remains persistence-agnostic.

---

## 7. Validation Pipeline

The validation pipeline must be deterministic and explicit.

## 7.1 Upload error check

The uploaded file must be checked for upload-level errors before any other processing step.

Any non-success upload state must fail immediately.

## 7.2 Source readability check

The processor must confirm that the underlying source stream or temp file is readable.

## 7.3 Max file size check

If `maxSizeBytes` is set and greater than zero, the uploaded video size must not exceed it.

### Important rule

`null` means no restriction.
The consuming application is responsible for normalizing zero-equivalent values consistently if desired.

## 7.4 Extension validation

If `allowedExtensions` is provided and non-empty, the source file extension must exist in the allowed set.

### Important rule

Extension validation is a policy check. It is not sufficient by itself to prove the file is a valid video.

## 7.5 MIME validation

If `allowedMimeTypes` is provided and non-empty, the actual video MIME type must match one of the allowed values.

### v1 rule

The implementation may use a safe MIME inspection strategy appropriate for video files, but it must not blindly trust browser-declared MIME.

The chosen MIME detection strategy must be implementation-documented and consistent across the module. Client-declared MIME alone is **never sufficient** as the sole acceptance signal.

If authoritative MIME detection cannot confirm the file appropriately, the processor must fail rather than silently accept ambiguity.

## 7.6 Corrupted file rejection

If the file is unreadable, malformed, or not a valid video according to the chosen detection strategy, processing must fail.

---

## 8. Temporary Preparation Rules

## 8.1 Single prepared output in v1

Version 1 produces exactly one prepared output file.

## 8.2 No transcoding in v1

The prepared file is not a transcoded or transformed video in version 1.

### Expected behavior

In v1, “prepared output” may mean a **controlled validated staged artifact**, not necessarily a transformed video file.

The processor validates the source and produces a controlled temporary output artifact representing the validated file.

## 8.3 Temp working directory

The processor may use:

* a temp directory explicitly supplied in the command DTO
* or a configured/default temporary working directory

## 8.4 Temp naming

The prepared temp file must use a deterministic safe naming strategy and must preserve the effective extension of the validated video unless an explicit future transformation policy changes it.

## 8.5 Temp cleanup on failure

If validation or preparation fails after temporary artifacts are created, the processor must clean them before throwing unless a documented retry-safe policy states otherwise.

---

## 9. Metadata Rules

Video metadata in version 1 is optional beyond core type and size information.

## 9.1 Required minimum metadata

The processor must return at minimum:

* `mimeType`
* `extension`
* `sizeBytes`

## 9.2 Optional metadata

If lightweight and safe to extract without introducing heavy runtime dependencies, the processor may additionally return:

* `durationSeconds`
* `width`
* `height`

### Important v1 rule

Advanced metadata extraction must not be mandatory in version 1.

If such metadata is unavailable, the result may expose `null` for those optional fields.

---

## 10. Output Contract

`VideoProcessor` must **never** return `StoredFileDTO` or any storage-layer success DTO.

The processor returns a formal process result DTO.

## 10.1 `VideoProcessResultDTO`

### Minimum fields

* `preparedFile`: `PreparedFileDTO`
* `metadata`: `VideoMetadataDTO`
* `warnings`: `array<int,string>`

### Rules

* `preparedFile.mediaType` must be `video`
* `warnings` must not be used to hide true validation or preparation failures

## 10.2 `PreparedFileDTO`

The returned prepared file must include at minimum:

* `mediaType = video`
* `tempPath`
* `finalExtension`
* `finalMimeType`
* `sizeBytes`
* `metadata = VideoMetadataDTO|null`
* `status = prepared`

## 10.3 `VideoMetadataDTO`

### Required minimum fields

* `mimeType`
* `extension`
* `sizeBytes`

### Optional fields

* `durationSeconds`
* `width`
* `height`

---

## 11. Exception Map

The exact inheritance tree may follow package conventions, but the following semantic exception set is required.

## 11.1 `VideoProcessorException`

Base video processor exception.

## 11.2 `VideoProcessorConfigurationException`

Thrown when required runtime configuration is missing or invalid.

## 11.3 `VideoValidationException`

Thrown when one or more validation rules fail.

### Important rule

This exception may expose a structured validation error bag.

## 11.4 `UnsupportedVideoMimeTypeException`

Thrown when the detected MIME type is not allowed.

## 11.5 `UnsupportedVideoExtensionException`

Thrown when the source extension is not allowed.

## 11.6 `VideoSizeViolationException`

Thrown when the source exceeds the allowed size.

## 11.7 `VideoReadException`

Thrown when the source cannot be read or safely interpreted as a valid video.

## 11.8 `VideoPreparationException`

Thrown when a valid video cannot be prepared into the required temporary output artifact.

## 11.9 `PreparedVideoCleanupException`

Thrown when cleanup after a failed preparation also fails.

---

## 12. Acceptance Criteria

The video processor is considered correct only if all of the following are true.

## 12.1 Validation correctness

* invalid upload states are rejected
* invalid MIME is rejected according to the configured detection strategy
* invalid extension is rejected when an extension policy exists
* oversized files are rejected
* corrupt or unreadable files are rejected

## 12.2 Preparation correctness

* valid videos produce a prepared temp file
* prepared MIME and extension match the validated source or documented v1 behavior
* returned metadata is internally consistent

## 12.3 Cleanup correctness

* failed preparation does not leave uncontrolled temp artifacts
* successful preparation returns a usable prepared temp file for handoff to storage

## 12.4 Architectural correctness

* no final storage is performed
* no database dependency exists
* no transcoding occurs in v1
* no poster generation occurs in v1

## 13.1 Validation tests

## 14. Usage Examples

## 14.1 Private Spaces video upload

The application receives a video upload and supplies:

* allowed extensions including `mp4`
* allowed MIME types including `video/mp4`
* a max size policy

The processor validates the upload, creates a prepared temp file, and returns a `PreparedFileDTO` with:

* `mediaType = video`
* `finalExtension = mp4`
* `finalMimeType = video/mp4`

The application then stores it using `StorageHandler` at a path such as:

* `actors/10/catalogs/15/video/source.mp4`

with private visibility.

## 14.2 Unsupported video type rejection

The application allows only `mp4` and `webm`.

A source file with a disallowed extension or MIME is rejected through a validation exception.

## 16. Final v1 Decisions

This specification makes the following decisions final for version 1.

1. `VideoProcessor` validates and prepares video files only.
2. It is fully database-agnostic.
3. It accepts rules through DTOs only.
4. It performs no final storage.
5. It performs no transcoding in v1.
6. It performs no poster or thumbnail generation in v1.
7. It returns exactly one prepared temporary output file.
8. Basic metadata is required; advanced metadata is optional.
9. Validation failures are exception-driven.
10. The prepared temp file is the downstream handoff unit to `StorageHandler`.

This specification is the authoritative v1 contract for `Maatify\VideoProcessor`.
