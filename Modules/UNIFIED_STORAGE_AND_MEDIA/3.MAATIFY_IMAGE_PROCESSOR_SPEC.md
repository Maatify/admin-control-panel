# MAATIFY Image Processor Specification

## 1. Purpose

`Maatify\ImageProcessor` is the dedicated module responsible for validating, inspecting, optionally transforming, and preparing image files for final storage.

It is the authoritative image-processing layer in version 1.

This module receives uploaded images and image-related rule DTOs from the consuming application, validates them using strict content-based inspection, optionally applies output-level transformations such as format conversion and quality optimization, and returns a prepared temporary file that is ready to be handed to `Maatify\StorageHandler`.

This module must remain fully independent from database persistence, admin panels, and storage backends.

---

## 2. Responsibilities

`ImageProcessor` is responsible for the following.

### 2.1 Content-based image validation

Validate the image using real binary inspection rather than trusting client-provided metadata.

### 2.2 Imagick-based image loading

Use Imagick as the image inspection and transformation engine.

### 2.3 Rule-based validation

Apply image rules carried by `ImageProfileDTO` or equivalent external DTOs.

### 2.4 Optional output transformation

Optionally convert the prepared output image format according to the supplied profile.

### 2.5 Optional optimization

Apply supported output quality optimization in a deterministic and documented way.

### 2.6 Prepared temporary output generation

Produce a temporary prepared output file and return it in a formal result DTO.

### 2.7 Metadata extraction

Return validated image metadata needed by the application or downstream storage workflows.

---

## 3. Explicit Non-Responsibilities

`ImageProcessor` must **not** do any of the following.

### 3.1 No final storage

It must not write to local storage roots or remote object stores as final persistence.

### 3.2 No business path generation

It must not decide target folder hierarchy or final business file names.

### 3.3 No direct database access

It must not read `maa_image_profiles` directly or any other table directly.

### 3.4 No image variants in v1

It must not generate multiple output files such as thumbnails, mobile variants, or alternate renditions in version 1.

### 3.5 No hidden orchestration

It must not silently decide storage drivers, storage visibility, or storage locations.

---

## 4. Architectural Position

`ImageProcessor` sits between raw uploaded input and final persistence.

### Flow

1. The consuming application receives a PSR-7 uploaded file.
2. The consuming application constructs `ImageProfileDTO` from any external source.
3. The application sends the command DTO to `ImageProcessor`.
4. `ImageProcessor` validates the source image and optionally transforms the output.
5. `ImageProcessor` returns a prepared temporary image file and metadata.
6. The application decides the final target path and hands the prepared file to `StorageHandler`.

This module therefore owns **image preparation**, not **image persistence**.

---

## 5. Required Runtime Dependencies

Version 1 requires the following runtime environment.

* PHP 8.1+
* `ext-imagick`
* `ext-bcmath`
* PSR-7 uploaded file support
* Maatify exception conventions

### Important runtime rule

If Imagick is unavailable or misconfigured, the module must fail explicitly through a configuration/runtime exception. Silent degradation is forbidden.

---

## 6. Input Model

`ImageProcessor` consumes normalized DTO-based input.

## 6.1 `ProcessImageCommandDTO`

### Minimum fields

* `source`: `UploadedSourceDTO`
* `profile`: `ImageProfileDTO`
* `tempDirectory`: `string|null`

### Purpose

Carry the uploaded source file and the external image rules into the processor.

---

## 6.2 `UploadedSourceDTO`

This DTO wraps the original uploaded file and request-level hints.

### Important rule

Values such as client filename or client media type are informational only. They are not authoritative validation sources.

---

## 6.3 `ImageProfileDTO`

This DTO carries image rules.

### Architectural rule

The processor does not know where this DTO came from.

The consuming application may build it from:

* `maa_image_profiles`
* a config file
* a static definition
* another service

The processor remains database-agnostic.

---

## 7. ImageProfileDTO Contract in v1

The following semantic fields are required in version 1.

* `code: string`
* `isActive: bool`
* `minWidth: int|null`
* `minHeight: int|null`
* `maxWidth: int|null`
* `maxHeight: int|null`
* `maxSizeBytes: int|null`
* `allowedExtensions: array<int,string>|null`
* `allowedMimeTypes: array<int,string>|null`
* `minAspectRatio: string|null`
* `maxAspectRatio: string|null`
* `requiresTransparency: bool`
* `preferredFormat: string|null`
* `preferredQuality: int|null`

### Important v1 rule

`variants` are intentionally excluded from the operational v1 image processing contract even if such data exists elsewhere.

---

## 8. Validation Pipeline

The validation pipeline must be strict and deterministic.

## 8.1 Upload error check

The uploaded file must first be checked for upload-level errors.

If the PSR-7 uploaded file reports a non-success upload state, processing must stop immediately.

## 8.2 Source file readability check

The module must verify that the uploaded source stream or temp file is actually readable before attempting Imagick inspection.

## 8.3 Profile active check

If `isActive` is false, the processor must reject the request.

## 8.4 Declared max file size check

If `maxSizeBytes` is set and greater than zero, the uploaded source size must not exceed it.

### Important rule

A `null` value means no restriction.
A zero-equivalent value must be treated consistently according to the consuming DTO normalization policy.

## 8.5 Binary image load via Imagick

The image must be loaded with Imagick for authoritative inspection.

If Imagick cannot read the file as a valid image, processing must fail.

## 8.6 MIME validation via Imagick

The image MIME type must be determined using Imagick inspection.

The processor must not trust:

* client media type
* browser-provided MIME
* extension alone

If `allowedMimeTypes` is provided and non-empty, the actual inspected MIME type must exist in the allowed list.

## 8.7 Extension validation

The extension may be derived from the source file name for policy checks, but the authoritative content inspection remains the MIME and the actual image load.

If `allowedExtensions` is provided and non-empty, the source extension must be allowed.

### Important rule

Extension validation is a policy check, not a substitute for real content validation.

## 8.8 Width and height validation

If profile width or height boundaries are provided, the actual image dimensions must satisfy them.

Applicable rules:

* `minWidth`
* `minHeight`
* `maxWidth`
* `maxHeight`

Any violation must be reported as a structured validation failure.

## 8.9 Aspect ratio validation

If `minAspectRatio` and/or `maxAspectRatio` are provided, the processor must compute the ratio as:

`width / height`

using `bcmath`-safe precision logic.

### Required comparison policy

* ratio precision must be normalized to 4 decimal places
* comparisons must be deterministic
* floating-point ambiguity must be avoided where practical

If the computed ratio falls outside the allowed range, processing must fail.

## 8.10 Transparency requirement validation

If `requiresTransparency` is true, the image must have a real alpha-capable state.

### Required check

The processor must inspect transparency using Imagick capabilities, not by extension name alone.

### Important v1 interpretation

In v1, the requirement is satisfied by a valid alpha-capable image state as recognized by Imagick. The processor is not required to prove that visible semi-transparent pixels are actively used in the content.

This keeps the rule strict but practical.

## 8.11 Corrupted image detection

If the image file is corrupt, truncated, malformed, or otherwise unreadable by Imagick, the processor must fail before any prepared output is produced.

---

## 9. Processing Rules

Once validation succeeds, the module may produce a processed output according to the profile.

## 9.1 No transformation when not requested

If `preferredFormat` is `null` and no other output transformation rules are active, the processor may preserve the original effective image format.

## 9.2 Preferred format conversion

If `preferredFormat` is provided, the processor should attempt to convert the image to that format, provided that:

* the target format is supported by the runtime
* the conversion is valid for the source image
* the operation can be completed safely

### Expected supported targets in v1

Support depends on runtime capabilities, but the architecture expects common support for:

* `jpg` / `jpeg`
* `png`
* `webp`

### Important rule

If the requested preferred format is unsupported in the current runtime, the processor must fail explicitly rather than silently ignoring the preference.

## 9.3 Preferred quality handling

If `preferredQuality` is provided, it must be applied only where the target format and Imagick behavior support meaningful quality control.

### Rules

* accepted range is expected to be normalized by the consuming DTO or enforced during processing
* quality handling must be deterministic and documented
* unsupported quality behavior must not silently corrupt the output

## 9.4 Metadata stripping policy

Version 1 must define one explicit metadata policy.

### Decision for v1

The processor may strip unnecessary metadata where safe and supported, but this behavior must be deterministic and documented in implementation.

If stripping is implemented, it must not remove data required for successful rendering.

## 9.5 EXIF orientation policy

The processor must define how orientation is handled.

### Decision for v1

The processor should normalize orientation before final output where the source format and runtime support it, so that width/height and rendered output remain consistent.

This behavior must be documented in code and tests.

---

## 10. Temporary Output Rules

## 10.1 Prepared output is always temporary

The processor must output a prepared temporary file, not a final business storage file.

## 10.2 Temp working directory

The processor may use:

* a provided temp directory from the command DTO
* or a configured/default temporary working directory

## 10.3 Operation isolation

The processor should isolate work by unique operation directory or unique temp file naming.

## 10.4 Output extension

The temporary file extension must match the final prepared format.

Examples:

* source `jpg`, output `webp` Ôćĺ temp file ends in `.webp`
* source `png`, no conversion Ôćĺ temp file may remain `.png`

## 10.5 Output MIME type

The returned prepared file must expose the actual final MIME type.

## 10.6 Temp cleanup on failure

If validation or processing fails after temporary artifacts are created, the processor must clean them before throwing, unless the implementation documents a different recovery-safe policy.

---

## 11. Output Contract

The processor returns a process result DTO carrying the prepared file and validated metadata.

## 11.1 `ImageProcessResultDTO`

### Minimum fields

* `preparedFile`: `PreparedFileDTO`
* `metadata`: `ImageMetadataDTO`
* `warnings`: `array<int,string>`

### Rules

* `preparedFile.mediaType` must be `image`
* `preparedFile.metadata` may duplicate `metadata`, but the top-level result must remain consistent
* `warnings` must not be used to hide actual validation or processing failures

## 11.2 `PreparedFileDTO`

The prepared file returned by the image processor must include at minimum:

* `mediaType = image`
* `tempPath`
* `finalExtension`
* `finalMimeType`
* `sizeBytes`
* `metadata = ImageMetadataDTO`
* `status = prepared`

## 11.3 `ImageMetadataDTO`

The image metadata must include at minimum:

* `width`
* `height`
* `mimeType`
* `extension`
* `sizeBytes`
* `hasTransparency`
* optional `aspectRatio`

---

## 12. Exception Map

The exact inheritance tree may be adjusted to package standards, but the following semantic exception set is required.

## 12.1 `ImageProcessorException`

Base image processor exception.

## 12.2 `ImageProcessorConfigurationException`

Thrown when required runtime dependencies or processor configuration are unavailable or invalid.

## 12.3 `InactiveImageProfileException`

Thrown when the provided profile is inactive.

## 12.4 `ImageValidationException`

Thrown when one or more validation rules fail.

### Important rule

This exception should expose a structured validation error bag.

## 12.5 `UnsupportedImageMimeTypeException`

Thrown when the inspected MIME type is not supported by the provided rules.

## 12.6 `UnsupportedImageExtensionException`

Thrown when the source extension violates the provided rules.

## 12.7 `ImageDimensionViolationException`

Thrown when min/max width or height validation fails.

## 12.8 `ImageAspectRatioViolationException`

Thrown when ratio validation fails.

## 12.9 `TransparencyRequirementViolationException`

Thrown when transparency is required but not satisfied.

## 12.10 `ImageReadException`

Thrown when the source cannot be read as a valid image.

## 12.11 `ImageProcessingFailureException`

Thrown when the source is valid but transformation or output generation fails.

## 12.12 `PreparedImageCleanupException`

Thrown when cleanup after a processing failure also fails.

---

## 13. Acceptance Criteria

The image processor is considered correct only if all of the following are true.

## 13.1 Validation correctness

* invalid upload states are rejected
* invalid MIME is rejected based on Imagick inspection
* invalid extension is rejected when extension policy is provided
* invalid dimensions are rejected
* invalid aspect ratio is rejected using deterministic precision logic
* missing required transparency is rejected
* corrupted images are rejected

## 13.2 Processing correctness

* valid images produce a prepared temp file
* requested supported format conversion works
* preferred quality is applied deterministically where supported
* output MIME and extension match actual prepared output
* returned metadata matches the prepared output

## 13.3 Cleanup correctness

* failed processing does not leave uncontrolled temp artifacts
* successful processing returns a usable prepared temp file for handoff

## 13.4 Architectural correctness

* no direct DB dependency exists
* no final storage is performed
* no path naming is imposed
* no variants are generated in v1

---

## 14. Testing Requirements

The image processor must be backed by automated tests covering the following.

## 14.1 Validation tests

* upload error handling
* MIME validation using actual image content
* extension validation
* min/max width validation
* min/max height validation
* max file size validation
* inactive profile rejection

## 14.2 Aspect ratio tests

* exact allowed ratio
* below minimum ratio
* above maximum ratio
* 4-decimal precision behavior

## 14.3 Transparency tests

* transparent PNG passes when required
* non-transparent JPEG fails when transparency is required
* alpha-capable valid image passes according to v1 rule

## 14.4 Processing tests

* preserve format when no conversion requested
* convert JPEG to WebP when requested
* convert PNG to WebP when requested
* quality handling for supported formats

## 14.5 Corruption and failure tests

* broken image input
* unreadable temp source
* unsupported preferred format
* cleanup after failed transform

## 14.6 Metadata tests

* width and height correctness
* MIME and extension correctness
* transparency metadata correctness

## 15. Usage Examples

## 15.1 Product image converted to WebP

The application receives a product image upload and builds an `ImageProfileDTO` whose preferred format is `webp`.

The processor validates the source image, converts it to WebP, writes a prepared temp file, and returns a `PreparedFileDTO` with:

* `finalExtension = webp`
* `finalMimeType = image/webp`

The application then chooses a final storage path such as:

* `catalogs/15/original.webp`

and passes the prepared file to `StorageHandler`.

## 15.2 Avatar image preserved as PNG

The application provides a profile with no preferred format conversion.

The processor validates the uploaded PNG and returns a prepared PNG file without conversion.

The application stores it under its chosen path.

## 15.3 Transparency-required image

The application provides a profile where `requiresTransparency = true`.

A JPEG upload fails validation.
A transparency-capable PNG or WebP passes if the runtime-inspected transparency requirement is satisfied according to v1 rules.

## 17. Final v1 Decisions

This specification makes the following decisions final for version 1.

1. `ImageProcessor` validates and prepares images only.
2. It is fully database-agnostic.
3. `ImageProfileDTO` is external-source driven.
4. Imagick is mandatory.
5. MIME validation is content-based.
6. Aspect ratio validation uses deterministic 4-decimal precision logic.
7. Transparency rules are enforced via Imagick-capable inspection.
8. Preferred format conversion is supported when runtime support exists.
9. No image variants are allowed in v1.
10. The module always returns a prepared temporary file for downstream storage.

This specification is the authoritative v1 contract for `Maatify\ImageProcessor`.
