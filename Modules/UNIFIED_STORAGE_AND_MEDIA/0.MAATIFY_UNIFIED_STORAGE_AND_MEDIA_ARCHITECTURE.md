# MAATIFY Unified Storage and Media Architecture

## 1. Purpose of This Architecture

This document defines the top-level architecture for the new Maatify media and storage ecosystem.

The goal is to replace loosely coupled legacy upload behavior with a strict, modular, DTO-driven architecture that separates:

* media validation and preparation
* media-type-specific processing
* final storage responsibilities
* business-level path ownership

This architecture is intentionally designed to avoid early over-engineering while still providing strong long-term extensibility.

It is the reference document that governs how the following modules work together:

* `Maatify\StorageHandler`
* `Maatify\ImageProcessor`
* `Maatify\VideoProcessor`
* `Maatify\AudioProcessor`

This document does **not** define every method or DTO in full depth. Those details belong to the dedicated specification of each module. This document defines the system boundaries, responsibility map, interaction model, and scope rules.

---

## 2. Design Goals

The architecture must satisfy the following goals.

### 2.1 Strict modularity

Each module must have a narrow, explicit responsibility.

### 2.2 DTO-driven integration

Modules must consume DTOs, commands, and contracts only. They must not depend directly on database tables or admin modules.

### 2.3 Low coupling

Storage must not know image rules. Image processing must not know storage internals. Business applications must own business path structure.

### 2.4 High extensibility

The system must support future expansion such as:

* image variants
  n- signed URLs
* advanced video processing
* advanced audio processing
* extra storage drivers

without forcing those features into version 1.

### 2.5 Low implementation cost

The first version must be practical to build, test, and maintain without introducing unnecessary abstraction complexity.

### 2.6 Strict error behavior

The system must use exceptions only. No silent failures, no `exit`, no `die`, no `header`, no array-based pseudo-results.

### 2.7 Business-path freedom

The system must not impose a fixed business path convention such as `media/images/...`. The consuming application is responsible for the final relative target path.

## 3. Version 1 Scope

Version 1 is intentionally strict and limited.

### 3.1 Included in v1

#### StorageHandler

* local storage support
* DigitalOcean Spaces support
* public/private visibility support
* safe target path validation
* final file persistence
* delete support
* exists support
* URL resolution where possible

#### ImageProcessor

* uploaded image validation
* Imagick-based MIME validation
* width and height validation
* file size validation
* aspect ratio validation
* transparency requirement validation
* optional output format conversion
* optional output optimization
* prepared temporary file output

#### VideoProcessor

* uploaded video validation
* extension and MIME checks
* file size validation
* prepared temporary file output

#### AudioProcessor

* uploaded audio validation
* extension and MIME checks
* file size validation
* prepared temporary file output

### 3.2 Explicitly excluded from v1

* image variants
* image batch outputs
* video transcoding
* audio transcoding
* waveform generation
* poster generation
* signed URLs
* direct database access
* direct dependency on `maa_image_profiles`
* forced naming strategy
* forced folder hierarchy

## 4. Core Modules Overview

## 4.1 `Maatify\StorageHandler`

This module is responsible only for final storage operations.

It accepts already-prepared files and stores them using a configured storage driver.

It does not validate image dimensions, convert image formats, or understand business models.

### Main responsibilities

* persist prepared files locally or remotely
* enforce safe target path rules
* map visibility to driver behavior
* resolve URLs where supported
* delete stored files
* support safe cleanup behavior

### Non-responsibilities

* image processing
* image optimization
* image profile interpretation
* video/audio preparation
* business path generation

---

## 4.2 `Maatify\ImageProcessor`

This module is responsible for validating and preparing image files.

It receives image-related DTOs from the outside world. These DTOs may originate from another module such as an image-profile module, but `ImageProcessor` must never depend directly on that external module's persistence layer.

### Main responsibilities

* inspect uploaded images using Imagick
* validate profile-driven rules
* optionally convert the image format
* optionally optimize output quality
* produce a prepared temporary output file

### Non-responsibilities

* final storage
* path selection
* business naming
* variant fan-out in v1

---

## 4.3 `Maatify\VideoProcessor`

This module is responsible for validating and preparing video files in version 1.

In v1, it is intentionally simple and does not include transcoding or preview generation.

### Main responsibilities

* validate supported video input
* validate file size
* prepare a temporary output file
* expose basic metadata when lightweight and safe

### Non-responsibilities

* transcoding
* thumbnails/posters
* HLS/DASH generation
* final storage

---

## 4.4 `Maatify\AudioProcessor`

This module is responsible for validating and preparing audio files in version 1.

In v1, it does not include transcoding, waveform generation, or normalization.

### Main responsibilities

* validate supported audio input
* validate file size
* prepare a temporary output file
* expose basic metadata when lightweight and safe

### Non-responsibilities

* transcoding
* waveform generation
* normalization
* final storage

---

## 5. System Boundaries

The system is intentionally split into three responsibility layers.

## 5.1 Consuming application layer

The consuming project is responsible for:

* choosing which processor to use
* constructing DTOs and command objects
* deciding the final relative target path
* deciding which storage driver and visibility should be used
* deciding whether a returned URL is required or whether a relative path is enough
* mapping business entities such as actor, catalog, product, tenant, or customer into folders and file names

Examples of valid application-owned path designs:

* `{catalog_id}/original.webp`
* `{catalog_id}/original.jpg`
* `{actor_id}/{catalog_id}/original.webp`
* `catalogs/{catalog_id}/gallery/main.webp`
* `tenants/{tenant_id}/actors/{actor_id}/audio/source.mp3`

The framework must not force these decisions.

## 5.2 Processor layer

Processor modules are responsible only for validating and preparing files.

They do not decide where the file will live permanently.

## 5.3 Storage layer

StorageHandler is responsible only for persisting a prepared file to the target path requested by the application.

It must not impose business semantics on that path.

---

## 6. End-to-End Workflow

## 6.1 Image upload workflow

1. The application receives an uploaded image.
2. The application builds an `ImageProfileDTO` or equivalent validation/options DTO.
3. The application sends the file and DTO to `ImageProcessor`.
4. `ImageProcessor` validates the file and prepares a temporary output file.
5. The application builds a `StorageRequestDTO` with:

   * driver
   * visibility
   * target relative path
6. The application sends the prepared file and storage request to `StorageHandler`.
7. `StorageHandler` stores the file and returns storage result DTOs.
8. The application decides what to persist in its own database.

## 6.2 Video upload workflow

1. The application receives an uploaded video.
2. The application builds a `VideoRulesDTO` or equivalent command DTO.
3. The application sends the file to `VideoProcessor`.
4. `VideoProcessor` validates and prepares a temporary file.
5. The application creates a storage request.
6. `StorageHandler` persists the file.
7. The application persists the returned storage metadata.

## 6.3 Audio upload workflow

1. The application receives an uploaded audio file.
2. The application builds an `AudioRulesDTO` or equivalent command DTO.
3. The application sends the file to `AudioProcessor`.
4. `AudioProcessor` validates and prepares a temporary file.
5. The application creates a storage request.
6. `StorageHandler` persists the file.
7. The application persists the returned storage metadata.

---

## 7. Data Flow Philosophy

The core rule is:

**processor modules prepare files, storage modules persist files, applications own business intent.**

This means:

* processors return prepared temporary artifacts
* storage accepts prepared artifacts and target instructions
* no module directly reads business tables
* no module assumes folder naming conventions from domain entities

The new ecosystem must be DTO-driven from start to finish.

---

## 8. Path Ownership Rules

## 8.1 The application owns the path

The final relative path is owned by the consuming project, not by the media modules.

## 8.2 The framework validates but does not invent business structure

The framework may validate the path for safety, but it must not generate domain-specific folder structure by assumption.

## 8.3 Target paths may be nested

Nested paths such as the following are valid:

* `15/original.webp`
* `10/15/original.webp`
* `actors/10/catalogs/15/original.webp`

## 8.4 The framework may offer optional helpers later

A future helper layer may provide convenience naming helpers, but v1 must not require them.

---

## 9. Error Handling Philosophy

## 9.1 Exceptions only

All operational failures must be represented by exceptions.

## 9.2 Validation failures must be structured

Validation exceptions must support a structured error bag where relevant.

## 9.3 No legacy response style

The new system must not return legacy-style arrays such as:

* `['uploaded' => 0, ...]`
* `['uploaded' => 1, ...]`

## 9.4 No unsafe flow termination

The new system must never use:

* `exit`
* `die`
* `header`
* error suppression via `@`

## 9.5 Resource cleanup is mandatory

Temporary files and processor resources must be cleaned up deterministically.

---

## 10. Temporary File Lifecycle

## 10.1 Temporary output is processor-owned until handoff

Processor modules create prepared temporary files.

## 10.2 Storage consumes prepared files

Once the application passes a prepared file to `StorageHandler`, the storage operation may consume or copy that file depending on the driver behavior.

## 10.3 Cleanup rules must be explicit

Each processor specification must define:

* where temporary files are created
* when they are deleted
* what happens on failure
* who owns cleanup after successful handoff

## 10.4 Ownership of successful cleanup

In v1, cleanup of prepared temp files after successful storage belongs to the application/workflow layer unless an explicit storage workflow contract states otherwise.

## 10.5 Operation isolation is recommended

Temporary work should be isolated by operation id or unique working directory where practical.

---

## 11. Shared Security Rules

## 11.1 MIME trust must be content-based where possible

For images, MIME validation must be derived from Imagick inspection, not user-provided metadata.

## 11.2 Extension checks are secondary to actual file inspection

Client extension is useful, but not authoritative by itself.

## 11.3 Path traversal must be blocked

Target path validation must prevent traversal attempts and invalid escape sequences.

## 11.4 Visibility is storage policy, not processor policy

Processors do not decide public/private behavior. StorageHandler and the application own that concern.

## 11.5 Private objects must not assume public URL availability

When a storage target is private, URL resolution may be unavailable or may return `null` in v1.

## 11.6 Explicit overwrite rule

Version 1 rejects silent overwrites by default. Any overwrite behavior requires an explicit future policy and is not assumed implicitly.

---

## 12. Non-Goals

The following are outside the goals of this architecture in v1:

* forcing all media types into one fake universal processor
* forcing the application to use one naming/path convention
* coupling processing to storage
* coupling processing to database tables
* supporting multi-variant image output from day one
* making advanced media processing mandatory for basic upload flows

---

### 13.1 Future image expansions

### 13.2 Future video expansions

### 13.3 Future audio expansions

### 13.4 Future storage expansions

* capability registry
* adapter-specific policies

---

## 14. Architectural Decision Summary

This architecture makes the following decisions final for version 1:

1. Storage is a standalone module.
2. Images, videos, and audio are processed by separate modules.
3. The application owns business path structure.
4. Database tables are not direct dependencies of processor modules.
5. DTOs and commands are the integration surface.
6. Image variants are postponed.
7. Video and audio advanced processing are postponed.
8. Exceptions are mandatory and structured.
9. Prepared temporary files are the handoff unit between processing and storage.

These decisions are the baseline for all implementation specifications that follow.
