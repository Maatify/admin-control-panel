# MAATIFY Shared Media Contracts

## 1. Purpose

This document defines the shared language, contracts, DTO conventions, enums, and lifecycle rules that are common across the Maatify storage and media ecosystem.

It exists to prevent contract drift between:

* `Maatify\StorageHandler`
* `Maatify\ImageProcessor`
* `Maatify\VideoProcessor`
* `Maatify\AudioProcessor`

This document is the canonical source for:

* shared DTO naming
* command naming
* common enum values
* prepared file handoff rules
* storage request/result structure
* validation error shape
* cleanup ownership rules

Module-specific rules belong in each module specification. This document defines only the common contract layer.

---

## 2. Global Naming Rules

## 2.1 DTO suffix

All DTO classes must end with `DTO`.

Examples:

* `PreparedFileDTO`
* `StorageRequestDTO`
* `ImageMetadataDTO`
* `MediaOperationResultDTO`

## 2.2 Interface suffix

All interfaces must end with `Interface`.

Examples:

* `StorageHandlerInterface`
* `ImageProcessorInterface`
* `PreparedFileCleanerInterface`

## 2.3 Command naming

Command-style request objects must end with `CommandDTO`.

Examples:

* `ProcessImageCommandDTO`
* `ProcessVideoCommandDTO`
* `StorePreparedFileCommandDTO`

## 2.4 Result naming

Objects returned as the formal result of an operation must end with `ResultDTO`.

Examples:

* `ImageProcessResultDTO`
* `StorageResultDTO`
* `MediaOperationResultDTO`

## 2.5 Exception naming

Shared and module-specific exception classes must end with `Exception`.

Validation exceptions may expose structured error bags.

## 3. Shared Enums

The exact PHP implementation may use native enums, value objects, or strict string constants depending on package constraints. The semantic values below are fixed.

## 3.1 `MediaTypeEnum`

Represents the logical media family.

Allowed values:

* `image`
* `video`
* `audio`

## 3.2 `VisibilityEnum`

Represents intended storage visibility.

Allowed values:

* `public`
* `private`

## 3.3 `StorageDriverEnum`

Represents the selected storage backend.

Allowed values in v1:

* `local`
* `do_spaces`

## 3.4 `PreparedFileStatusEnum`

Represents the lifecycle state of a prepared file.

Allowed values:

* `prepared`
* `stored`
* `cleaned`
* `failed`

This enum is conceptual and may be internal if not exposed in the public API.

## 4. Shared DTO Contracts

## 4.1 `UploadedSourceDTO`

Represents the input file received from the consuming application before processing.

### Purpose

Provide a normalized contract around an uploaded file and any lightweight request context that processors may need.

### Minimum fields

* `uploadedFile`: `UploadedFileInterface`
* `originalClientFilename`: `string|null`
* `clientMediaType`: `string|null`
* `declaredSizeBytes`: `int|null`

### Rules

* `clientMediaType` is informational only.
* `originalClientFilename` is informational only.
* processors must not trust these values as authoritative content validation.

---

## 4.2 `PreparedFileDTO`

Represents a file that has already passed processing and is ready to be stored.

### Purpose

This is the canonical handoff unit from any processor module to `StorageHandler`.

### Minimum fields

* `mediaType`: `MediaTypeEnum`
* `tempPath`: `string`
* `finalExtension`: `string`
* `finalMimeType`: `string`
* `sizeBytes`: `int`
* `metadata`: `ImageMetadataDTO|VideoMetadataDTO|AudioMetadataDTO|null`
* `status`: `PreparedFileStatusEnum`

### Required guarantees

* `tempPath` must point to a real prepared file at handoff time.
* `finalExtension` must represent the actual prepared output extension.
* `finalMimeType` must represent the actual prepared output MIME type.
* `sizeBytes` must represent the prepared file size, not the original uploaded size unless unchanged.

### Rules

* This DTO must not contain final business storage path information.
* This DTO must not contain storage-driver-specific state.
* This DTO must be safe to pass between modules.

---

## 4.3 `StorageRequestDTO`

Represents the consumer's final storage intention.

### Purpose

Allows the application to instruct `StorageHandler` where and how the prepared file should be persisted.

### Minimum fields

* `driver`: `StorageDriverEnum`
* `visibility`: `VisibilityEnum`
* `targetRelativePath`: `string`

### Optional fields

* `diskKey`: `string|null`
* `resolveUrl`: `bool`
* `metadataHeaders`: `array<string,string>`

### Rules

* `targetRelativePath` is application-owned.
* the storage layer validates safety only; it does not invent business paths.
* `diskKey` is optional and reserved for future multi-disk resolution strategies.
* `metadataHeaders` is optional and adapter-dependent.

---

## 4.4 `StoredFileDTO`

Represents the final persisted file after successful storage.

### Minimum fields

* `mediaType`: `MediaTypeEnum`
* `driver`: `StorageDriverEnum`
* `visibility`: `VisibilityEnum`
* `relativePath`: `string`
* `resolvedUrl`: `string|null`
* `sizeBytes`: `int`
* `finalMimeType`: `string`
* `finalExtension`: `string`

### Rules

* `relativePath` is the canonical path to persist in application databases if the application chooses to store only one path value.
* `resolvedUrl` may be `null` for private storage in v1.
* `StoredFileDTO` must not expose adapter secrets or transport internals.

---

## 4.5 `MediaOperationResultDTO`

Represents the top-level result returned to the application after a complete process-and-store workflow, if a facade or orchestration layer is used.

### Minimum fields

* `storedFile`: `StoredFileDTO`
* `preparedFile`: `PreparedFileDTO|null`
* `warnings`: `array<int,string>`

### Rules

* This DTO is optional at the architecture level and may be omitted if the application directly calls processor and storage modules separately.
* If used, it must remain a thin orchestration result, not a place for hidden storage internals.

---

## 4.6 `ValidationErrorDTO`

Represents a single validation error entry.

### Minimum fields

* `field`: `string`
* `rule`: `string`
* `message`: `string`
* `context`: `array<string,mixed>`

### Examples

* field: `mime_type`

* rule: `allowed_mime_types`

* message: `Unsupported image MIME type.`

* field: `width`

* rule: `min_width`

* message: `Image width is below minimum allowed value.`

### Rules

* validation bags must contain structured entries, not just raw strings.
* `context` must be safe for debugging and error reporting.
* sensitive secrets must never be included.

## 6. Shared Rules DTOs

## 6.1 `ImageProfileDTO`

Represents externally supplied image validation and processing rules.

### Important architectural rule

This DTO may be constructed from `maa_image_profiles` or any other source, but the processor must not know or care where it came from.

### Minimum fields

* `code`: `string`
* `isActive`: `bool`
* `minWidth`: `int|null`
* `minHeight`: `int|null`
* `maxWidth`: `int|null`
* `maxHeight`: `int|null`
* `maxSizeBytes`: `int|null`
* `allowedExtensions`: `array<int,string>|null`
* `allowedMimeTypes`: `array<int,string>|null`
* `minAspectRatio`: `string|null`
* `maxAspectRatio`: `string|null`
* `requiresTransparency`: `bool`
* `preferredFormat`: `string|null`
* `preferredQuality`: `int|null`

### Rules

* decimal ratio values should be represented safely for precise comparison logic.
* `preferredFormat` is advisory to the processor and only applies when supported.
* image variants are excluded from v1 and therefore must not appear in this shared DTO contract for v1.

---

## 6.2 `VideoRulesDTO`

Represents externally supplied video validation rules.

### Minimum fields

* `allowedExtensions`: `array<int,string>|null`
* `allowedMimeTypes`: `array<int,string>|null`
* `maxSizeBytes`: `int|null`

### Rules

* no transcoding policy belongs here in v1.
* future duration or codec rules may be added later.

---

## 6.3 `AudioRulesDTO`

Represents externally supplied audio validation rules.

### Minimum fields

* `allowedExtensions`: `array<int,string>|null`
* `allowedMimeTypes`: `array<int,string>|null`
* `maxSizeBytes`: `int|null`

### Rules

* no transcoding policy belongs here in v1.
* future duration or bitrate rules may be added later.

## 8. Shared Exception Contract Rules

## 8.1 Exceptions are mandatory

All failures must be expressed via exceptions.

## 8.2 Validation exceptions may expose structured error bags

Any validation-focused exception may expose:

* `getErrors(): array<int,ValidationErrorDTO>`

or an equivalent structured shape.

## 8.3 Context must be safe

Exception context must never expose secrets such as access keys, secrets, or internal credentials.

## 8.4 Shared exception categories

The exact inheritance tree may differ by package, but the semantic groups are fixed:

* validation exceptions
* processing exceptions
* storage exceptions
* cleanup exceptions
* configuration exceptions

## 8.5 No legacy silent failure contract

Modules must not return booleans or arrays to indicate soft failure where an exception is required.

---

## 9. Shared Cleanup Rules

## 9.1 Processor-created temp files must be tracked

Every prepared file must have clear ownership and cleanup expectations.

## 9.2 Handoff changes ownership expectations

Once a prepared file is successfully handed to storage and persistence is completed, cleanup responsibility may move depending on implementation details documented in each module.

## 9.3 Failed processing must clean processor temp artifacts

If a processor fails after creating temporary artifacts, it must clean them before rethrowing unless a documented exception strategy says otherwise.

## 9.4 Failed storage must not leave ambiguous temp ownership

Storage failure paths must clearly state whether the prepared temp file remains available for retry or is cleaned immediately.

## 9.5 Cleanup must be idempotent where practical

Repeated cleanup calls should not create additional failures unless a real filesystem or adapter error occurs.

---

## 10. Shared Path and Filename Rules

## 10.1 Business path belongs to the application

Business path semantics are never defined by the shared contracts layer.

## 10.2 Relative paths only

Target paths in storage requests must be relative paths.

## 10.3 Unsafe path patterns are forbidden

Forbidden patterns include, but are not limited to:

* parent traversal such as `../`
* absolute path escapes when relative paths are required
* null byte tricks
* unsafe separator abuse

## 10.4 File naming is application-owned unless a processor converts extension

The application chooses the intended file name, while the processor may change the final extension only when processing rules require it.

Example:

* application intends `catalog/15/original.jpg`
* image processor converts the file to WebP
* final target path used by the application should therefore be adjusted to `catalog/15/original.webp`

This coordination responsibility belongs to the consuming application or any explicit orchestration layer.

---

## 11. Shared Architectural Rules

## 11.1 No direct DB dependency

Shared contracts must never require a direct database dependency.

## 11.2 No direct legacy `$_FILES` dependency

Public contracts must operate on normalized DTOs and PSR-7 file abstractions, not raw global arrays.

## 11.3 No hidden adapter-specific state in DTOs

Shared DTOs must remain portable.

## 11.4 No variants in v1 shared contracts

Variant-related DTOs are intentionally excluded from v1.

---

## 12. Summary of Final Shared Decisions

This document makes the following shared contract decisions final for version 1:

1. All DTO classes end with `DTO`.
2. All interfaces end with `Interface`.
3. All command request DTOs end with `CommandDTO`.
4. `PreparedFileDTO` is the canonical processor-to-storage handoff unit.
5. `StorageRequestDTO` is application-owned and path-driven.
6. `StoredFileDTO` is the canonical storage success result.
7. Validation errors must be structured.
8. `ImageProfileDTO` is external-source driven and database-agnostic.
9. Video and audio rules are simple in v1.
10. Variants are excluded from v1 shared contracts.

These shared rules must be honored by all module specifications that follow.
