# MAATIFY Storage Handler Specification

## 1. Purpose

`Maatify\StorageHandler` is the dedicated module responsible for final file persistence.

Its role is strictly limited to storage concerns.

It receives already-prepared files from processor modules or directly from application-level workflows and persists them using a configured storage backend.

This module is the single source of truth for:

* local file persistence
* DigitalOcean Spaces persistence
* visibility mapping (`public` / `private`)
* safe target path validation
* delete and existence checks
* URL resolution when supported
* storage-layer cleanup behavior

It must not contain image-specific, video-specific, or audio-specific business logic.

---

## 2. Responsibilities

`StorageHandler` is responsible for the following.

### 2.1 Final persistence

Persist a prepared file into the selected storage backend.

### 2.2 Safe path validation

Validate the requested relative path before writing.

### 2.3 Visibility enforcement

Map logical visibility (`public` / `private`) into backend-specific storage behavior.

### 2.4 URL resolution

Return a resolved URL when the storage driver and visibility settings allow it.

### 2.5 Deletion

Delete previously stored files using their canonical relative path.

### 2.6 Existence checks

Provide existence checks for relative stored paths.

### 2.7 Storage cleanup semantics

Define what happens when a write operation fails partially or when cleanup is required.

---

## 3. Explicit Non-Responsibilities

`StorageHandler` must **not** do any of the following.

### 3.1 No media validation

It must not validate image dimensions, aspect ratios, transparency, codecs, or media quality rules.

### 3.2 No media processing

It must not optimize, convert, compress, crop, transcode, or normalize files.

### 3.3 No direct database usage

It must not read database tables or know about business storage schema stored in databases.

### 3.4 No business path invention

It must not invent domain-specific folder structures such as product, actor, tenant, or catalog paths.

### 3.5 No storage-time filename generation by business assumption

It may validate a target file path, but the application owns the target path and file name.

---

## 4. Architectural Position

`StorageHandler` sits after processors and before database persistence at the application layer.

### Flow

1. A processor or application prepares a file.
2. The consuming application decides the final storage request.
3. `StorageHandler` validates the request and persists the file.
4. The application stores `StoredFileDTO` values in its own database if needed.

`StorageHandler` is therefore a pure infrastructure module.

---

## 5. Input Model

`StorageHandler` accepts the following shared contracts.

## 5.1 `PreparedFileDTO`

This is the file already validated and ready for persistence.

Required guarantees at handoff time:

* `tempPath` exists
* `finalMimeType` is accurate
* `finalExtension` is accurate
* `sizeBytes` matches the prepared output

## 5.2 `StorageRequestDTO`

This describes the consumer's storage intention.

Required fields:

* `driver`
* `visibility`
* `targetRelativePath`

Optional fields may include:

* `diskKey`
* `resolveUrl`
* `metadataHeaders`

---

## 6. Public API

The exact implementation may vary, but the semantic API for v1 is fixed.

## 6.1 `storePreparedFile()`

### Purpose

Store one prepared file in the selected backend.

### Input

* `StorePreparedFileCommandDTO`

### Output

* `StorageResultDTO`

### Behavior

* validate prepared file state
* validate target path safety
* resolve target adapter
* write file
* optionally resolve URL
* return canonical stored result

---

## 6.2 `deleteStoredFile()`

### Purpose

Delete a single stored file by canonical relative path.

### Input

* `StorageDeleteRequestDTO` or equivalent minimal request DTO

### Output

* `void`

### Behavior

* validate request
* delete if present
* ignore missing file where policy allows
* throw on actual backend deletion failure

---

## 6.3 `exists()`

### Purpose

Check whether a stored relative path currently exists.

### Input

* driver identifier
* relative path

### Output

* `bool`

---

## 6.4 `resolveUrl()`

### Purpose

Resolve a URL for a stored file when the backend and visibility allow it.

### Output

* `string|null`

### Rule

In v1, private objects may legitimately return `null`.

---

## 7. Driver Model

Version 1 supports exactly two storage drivers.

## 7.1 Local driver

Used for storage on the local filesystem.

### Responsibilities

* write files under a configured root
* create missing parent directories safely
* validate that final writes remain inside the configured root
* delete files safely
* resolve a public URL only when configured

## 7.2 DigitalOcean Spaces driver

Used for object storage via the S3-compatible API.

### Responsibilities

* upload the prepared file body to the configured bucket
* map visibility into object ACL or equivalent policy
* delete objects by key
* check object existence
* resolve public URL when supported

---

## 8. Visibility Model

## 8.1 Meaning of `public`

The consuming application intends the object to be retrievable through a non-secret URL.

### Local behavior

This typically means the file is placed in a web-accessible location or is resolvable through a configured public base URL.

### Spaces behavior

This typically means the object is written using public-read semantics or any equivalent policy configured by the driver.

## 8.2 Meaning of `private`

The consuming application intends the file to be non-public by default.

### Local behavior

The file may still exist on local disk but may not have a public URL mapping.

### Spaces behavior

The object must be uploaded as private unless the backend policy layer overrides this explicitly.

## 8.3 URL resolution rule

`public` does not guarantee that a URL must always be returned if the driver is not configured with a public resolver.

`private` in v1 may return `null` for URL resolution.

---

## 9. Path Rules

## 9.1 Relative paths only

`targetRelativePath` must always be a relative path.

Examples of valid target paths:

* `15/original.webp`
* `10/15/original.jpg`
* `actors/10/catalogs/15/audio/source.mp3`

## 9.2 Forbidden path patterns

The storage module must reject unsafe target paths including, but not limited to:

* parent traversal such as `../`
* absolute paths such as `/var/www/file.jpg`
* Windows-style root escapes when relative paths are required
* null bytes
* repeated ambiguous separators where they create unsafe normalization

## 9.3 Canonicalization rule

Path validation must normalize separators and validate the canonical relative result before writing.

## 9.4 Business semantics are not interpreted

A path like `actors/10/catalogs/15/original.webp` is treated only as a valid relative path, not as a business structure the module understands.

---

## 10. Local Driver Behavior

## 10.1 Config contract

The local driver must be configured with at least:

* storage root path
* optional public base URL

## 10.2 Directory creation

Missing parent directories must be created recursively when safe and necessary.

## 10.3 Root confinement

All writes must remain inside the configured local storage root.

## 10.4 Write behavior

The driver may copy, rename, or stream the prepared file into the final location depending on the implementation, but the behavior must be deterministic and documented.

## 10.5 Overwrite behavior

Version 1 must define one clear overwrite policy.

### Decision for v1

If the target path already exists, the write must fail by default unless an explicit future overwrite policy is introduced.

This prevents accidental silent replacement.

## 10.6 Delete behavior

Deleting a missing local file may be treated as a no-op.

Deleting an existing local file that cannot be removed is an error.

## 10.7 URL behavior

If a public base URL is configured and the visibility is public, the driver may return a resolved URL.

If no public base URL is configured, URL resolution may return `null`.

---

## 11. DigitalOcean Spaces Driver Behavior

## 11.1 Config contract

The Spaces driver must be configured with at least:

* access key
* secret key
* region
* endpoint
* bucket
* optional CDN/public base URL

## 11.2 Object key mapping

The final object key is the validated `targetRelativePath`.

## 11.3 Write behavior

The driver uploads the prepared file body to the target object key.

## 11.4 Content type

The upload request must use the prepared file MIME type as the object content type whenever supported.

## 11.5 Visibility mapping

The driver maps `public` or `private` into backend-specific object settings.

## 11.6 Delete behavior

Deleting a missing object may be treated as a no-op where backend semantics allow it.

## 11.7 Exists behavior

Object existence checks must use backend-supported existence operations.

## 11.8 URL behavior

If a public base URL or CDN URL is configured and the object is public, the driver may return a resolved URL.

If the object is private, URL resolution may return `null` in v1.

### Important v1 rule

Signed URLs are outside scope.

---

## 12. Prepared File Handling Rules

## 12.1 Prepared file must exist

Before writing, the storage layer must verify that the prepared temp file exists and is readable.

## 12.2 Prepared file is treated as source of truth

The storage layer must trust the processor-provided output artifact, not re-run media-specific validation.

## 12.3 MIME and extension

The storage layer uses the prepared file MIME and extension as authoritative metadata for storage operations.

## 12.4 Target path and prepared extension coordination

The consuming application is responsible for ensuring that the final `targetRelativePath` matches the prepared file's final extension.

Example:

* processor outputs `webp`
* application must request a target path ending in `.webp`

The storage layer may reject obvious extension mismatches if the implementation chooses to enforce this, but this is optional in v1.

---

## 13. Temporary File Ownership and Cleanup

## 13.1 Before successful write

Before successful storage, the prepared temp file remains a source artifact owned by the process workflow.

## 13.2 After successful write

After successful storage, the implementation must define a deterministic policy for the prepared temp file.

### Decision for v1

After successful persistence in v1, prepared temp file cleanup belongs to the **application/workflow layer by default**. `StorageHandler` does **not** implicitly delete the prepared temp file unless a future explicit contract adds that behavior.

The policy must be consistent and documented in code.

## 13.3 On write failure

If storage fails before persistence completes, the temp file should remain available unless the implementation intentionally cleans it as part of deterministic rollback.

This behavior must be explicit.

---

## 14. Atomicity and Rollback

Version 1 storage is single-file oriented.

Therefore, rollback behavior is simpler than future batch or variant storage.

## 14.1 Single-file storage atomicity

A storage operation is considered successful only when the final file or object is fully persisted and the module can return a valid `StoredFileDTO`.

## 14.2 Local partial failure

If a local write fails during copy or move, the implementation must ensure that corrupted partial output is not left behind whenever practical.

## 14.3 Remote partial failure

If a remote upload fails, the implementation must treat the operation as failed and must not return a success result.

If a partially created remote object exists and can be safely deleted, cleanup should be attempted.

## 14.4 Cleanup failure reporting

If cleanup after a failed write also fails, the thrown exception should carry both:

* the original write failure context
* the cleanup failure context

---

## 15. Result Contracts

## 15.1 `StorageResultDTO`

Represents a successful storage outcome.

`StorageResultDTO` is the operation result wrapper. `StoredFileDTO` is the canonical persisted-file payload inside it.

### Minimum fields

* `storedFile`: `StoredFileDTO`
* `cleanupPerformed`: `bool`
* `warnings`: `array<int,string>`

### Notes

* `cleanupPerformed` may refer to temp-file cleanup if the storage flow owns that behavior.
* `warnings` must not be used to hide actual failures.

---

## 16. Exception Map

The exact inheritance tree may be adjusted to package standards, but the following semantic exception types are required.

## 16.1 `StorageHandlerException`

Base storage module exception.

## 16.2 `InvalidTargetPathException`

Thrown when the requested relative path is unsafe or invalid.

## 16.3 `PreparedFileNotFoundException`

Thrown when the provided temp file path does not exist at handoff time.

## 16.4 `PreparedFileUnreadableException`

Thrown when the prepared temp file cannot be read.

## 16.5 `UnsupportedStorageDriverException`

Thrown when the selected driver is unknown or unavailable.

## 16.6 `StorageWriteException`

Thrown when final persistence fails.

## 16.7 `StorageDeleteException`

Thrown when deletion of an existing stored file fails.

## 16.8 `StorageExistsCheckException`

Thrown when an existence check fails at the backend layer.

## 16.9 `StorageUrlResolutionException`

Thrown when URL resolution was explicitly requested and fails in an invalid or unexpected way.

## 16.10 `StorageRollbackException`

Thrown when cleanup after a failed write also fails.

## 16.11 `StorageConfigurationException`

Thrown when required driver configuration is missing or invalid.

---

## 17. Acceptance Criteria

The implementation is considered correct only if all of the following are satisfied.

## 17.1 Local storage

* a valid prepared file is persisted under the requested relative path
* directories are created safely
* path traversal is blocked
* overwrites are not allowed silently
* public URL resolution works when configured
* private URL resolution may return `null`

## 17.2 Spaces storage

* a valid prepared file is uploaded under the requested object key
* content type is correct
* visibility mapping is honored
* private URL resolution may return `null`
* public URL resolution works when configured

## 17.3 Cleanup behavior

* deletion works for valid stored files
* deletion of missing files does not incorrectly fail unless policy requires it
* cleanup failures surface as exceptions

## 17.4 Contract behavior

* no media-specific validation occurs in the storage layer
* no direct DB dependency exists
* only DTO-based handoff is used

## 19. Usage Examples

## 19.1 Local public storage

A product image is processed to WebP and then stored at:

* `catalogs/15/original.webp`

The application sends a `PreparedFileDTO` with final extension `webp` and a `StorageRequestDTO` targeting that relative path using local public storage.

## 19.2 Spaces private storage

A video file is validated and prepared, then stored at:

* `actors/10/catalogs/15/video/source.mp4`

The application chooses `do_spaces` with private visibility.

The stored result returns:

* canonical relative path
* `resolvedUrl = null`

## 19.3 Nested business path

An audio file is stored at:

* `tenants/3/actors/99/catalogs/20/audio/source.mp3`

The storage layer treats this only as a valid relative path and does not attempt to interpret it.

## 20. Future Extensions

The following may be added later without changing the v1 responsibility model.

* batch store support
* signed URLs
* temporary URLs
* overwrite policies
* adapter capability registry
* stream-native persistence APIs
* multi-disk resolution via `diskKey`

---

## 21. Final v1 Decisions

This specification makes the following decisions final for version 1.

1. `StorageHandler` stores files only.
2. The consuming application owns the final relative path.
3. Only `local` and `do_spaces` are supported in v1.
4. Visibility is modeled as `public` or `private`.
5. Private storage may legitimately return `null` for URLs.
6. Overwrites are rejected by default.
7. No media-specific validation is allowed in the storage layer.
8. Prepared temp files are the handoff input.
9. Signed URLs are outside v1 scope.
10. Storage remains fully database-agnostic.

This specification is the authoritative v1 contract for `Maatify\StorageHandler`.
